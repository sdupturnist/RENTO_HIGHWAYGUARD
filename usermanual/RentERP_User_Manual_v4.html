System Overview & Core Operational Pipeline

An introduction to RentERP's operational core and how data flows through the fleet management cycle.

1.1 The Operational Pipeline

RentERP is designed to streamline vehicle and equipment rental workflows. The system maps your physical fleet operations directly to billing workflows through a simple, four-step pipeline:

1
Create Master Records

Configure active vehicles, operator details, customers, and project records in the database.

2
Deploy Assignments

Schedule resources to projects by creating vehicle, operator, material, or detour blocks.

3
Log Daily Work (DTLs)

Record working hours, holiday flags, and shift overrides for each deployed resource.

4
Generate & Bill

Compile daily logs into approved timesheets and generate customer-facing invoices in one click.

Immutable Rule This pipeline is enforced by the system. You cannot create an invoice without a timesheet. You cannot create a timesheet without DTLs. DTLs cannot exist without an assignment block. The DTL is the single source of truth — every report, every utilization metric, every cost calculation traces back to it. Never collapse DTLs into summaries.

1.2 Resource Types

The system supports four distinct operational resources, each with custom billing and rate models:

  • Vehicles: Mobile assets (trucks, SUVs) or machinery (excavators, cranes) billed at daily or hourly rates.
  • Operators: Drivers or crew members deployed standalone or linked directly to vehicles.
  • Materials: Inventory items (barriers, generators) billed by quantity deployed per day.
  • Labour: General crew roles (helpers, riggers) billed as standard day shifts.

1.3 Assignment Block Types

When deploying resources, coordinators organize them in four block types:

  • VEHICLE Block: Schedules a machine to a project. Can be marked "With Operator" to automatically request operator rates, or "Without Operator" (dry hire).
  • OPERATOR Block: Standalone crew deployment (e.g., placing an operator to drive a customer-owned vehicle).
  • MATERIAL Block: Deploys a specific quantity of materials (e.g., deploying 50 safety blocks).
  • LABOUR Block: Deploys a specific headcount of crew roles.

Key Principle — DETOUR blocks A DETOUR block is purely a grouping mechanism. It never generates a daily time log directly. When the scheduler runs, it skips DETOUR blocks entirely. Each child VEHICLE / OPERATOR / MATERIAL / LABOUR block under the detour generates its own individual DTL.

Landing Dashboard, Quick Actions & Expiry Alerts

An overview of the central mission-control screen where system status, live clock, KPIs, quick shortcuts, and document expiration snapshots are displayed.

2.1 Core KPI Metric Cards

The landing dashboard loads a real-time summary of the system's operational and financial status. These are powered by cached aggregate database queries to ensure instantaneous loading:

  • Total Active Vehicles: Counts all vehicles whose status is set to ACTIVE. Provides a direct link to the filtered Vehicles List.
  • Active Assignments: Shows the number of vehicle/operator deployment blocks currently running on their calendar schedule.
  • Pending Timesheets: Summarizes the number of timesheets currently in DRAFT or SUBMITTED status awaiting customer billing approval.
  • Unpaid Invoices: Displays the count and total monetary value of invoices in SENT or PARTIALLY_PAID status, keeping outstanding Accounts Receivable (AR) highly visible.

2.2 The Interactive Dashboard Clock

A live-updating micro-widget in the header bar of the dashboard displays the active tenant's local time (configured in Settings → Localization). Since background DTL logs are generated according to this localized timezone, this clock helps operators instantly verify the exact hour boundary for scheduler runs and manual log creations.

2.3 Quick Action Center

To reduce click paths and improve operator throughput, the dashboard features a floating "Quick Actions" dashboard panel containing direct links to key creation dialogs:

  • Create Assignment: Launches the assignment generation flow directly from the dashboard, pre-selecting the current date.
  • Generate Timesheet: Prompts the operator for Customer, Project, and Billing Period, taking them directly to the timesheet editor.
  • Log Vehicle Maintenance: Pre-populates the vehicle selection dropdown and lets administrators quickly schedule corrective or breakdown repairs.

2.4 Expiry Alerts & Missing Documents Snapshot

An intelligence widget automatically scans all active vehicles and operators to surface compliance risks. The expiry panel on the dashboard categorises all tracked items into three urgency tiers:

Document Expiry Tiers:
  • EXPIRED: Expiration date is in the past. Immediate action required.
  • URGENT (≤ 7 days): Expiry occurs within the next 7 days. High-priority alert.
  • UPCOMING (> 7 days): Expiry is approaching but not yet critical. Visible for advance planning.

Documents tracked per vehicle: Registration, Insurance.

Documents tracked per operator: Visa, Driving Licence, Emirates ID, Medical Certificate.

Panel Only Appears When Enabled The expiry panel respects the enableExpiryNotifications toggle in Notification Settings. If disabled, the panel returns active: false and no items are shown on the dashboard.

System Settings & Customizations

A detailed user guide on how to configure your company profile, white-label branding, localized preferences, email servers, and operational billing rules.

3.1 Company Settings

Inside Settings → Company, administrators define the primary identity of their tenant organization. This profiles your legal name, physical address, phone contact, tax registration numbers, and official customer-facing email address. These details are dynamically loaded onto invoice headers and email footers.

Example — how Full Day Hours flows through everything If Full Day Hours = 8 and a vehicle block has plannedOvertimeHours = 2:
• DTL is created with workedHours = 10, regularHours = 8, overtimeHours = 2
• A DAILY-rate vehicle at AED 400/day converts to AED 50/hour (400 ÷ 8)
• Overtime: 2 hrs × AED 50 × 1.5 multiplier = AED 150 extra on the timesheet line

3.2 Branding Settings (White-Labeling)

To deliver a personalized experience for your team and clients, the system supports custom white-label branding:

  • Dashboard Logo: Upload a high-resolution logo (PNG or SVG) to replace the default app icon in the sidebar and header.
  • Brand Color Palette: Select your organization's primary color. The system dynamically updates navigation borders, active buttons, selection highlights, and email themes to match your brand style.
  • Company Slogan: Displays as a small subtitle under your company name in the sidebar.

3.3 Localization Settings

Align the application with your regional requirements:

  • Timezone: Select your local timezone. This controls the active clock on the dashboard and guarantees that auto-generated time logs are stamped accurately relative to your local work hours.
  • Date Format: Choose how dates appear across grids and documents (e.g. DD/MM/YYYY or MMM DD, YYYY).
  • Distance & Volume Metrics: Toggle default metrics (e.g. Kilometers vs Miles; Liters vs Gallons) for fuel logs and vehicle odometer tracking.

3.4 Email & Notification Settings

Configure communications in one central screen:

  • Email Server (SMTP): Enter your official email host, port, username, and password. The system uses these settings to automatically dispatch digital timesheets, invoice notifications, and daily expiry reminders.
  • Notification Toggles: Choose which alerts to enable. You can toggle off-hours notifications, enable dashboard-only alerts, or configure email dispatches for document expirations.

A "Test Connection" button sends a test email to verify the SMTP configuration before going live.
If SMTP is not configured and the system tries to send an email, the email operation fails gracefully. The underlying record (assignment, timesheet, invoice) is still saved successfully. Only the email send fails. The failure is logged but does not block the main operation.

3.5 Billing & Invoice Settings

Define standard business rules applied to all client accounts:

  • Grace Period Thresholds: Define a buffer (e.g., 15 minutes) for worked shifts before overtime rates are calculated.
  • Overage Rounding Rules: Set whether logged fractions of an hour round up, round down, or round to the nearest quarter-hour for billing.
  • Default Payment Terms: Set the system-wide billing default (e.g., Cash on Delivery, Net 30, Net 60) for new customer records.

3.6 Security & Safety Parameters

Protect your system from unauthorized access:

  • Password Strength: Select standard, medium, or high requirements for employee login passwords.
  • Account Lockout Limits: Specify the number of failed attempts allowed before a user account is locked out, helping block automated entry attempts.
  • Automatic Session Timeout: Set inactivity limits to automatically log out users if a computer is left unattended on site.

Master Data Portals

A guide to the primary records of the application: Vehicles, Operators, Customers, Projects, Materials, and Labours.

5.1 Fleet Vehicle Portal (`/vehicles`)

The Fleet Vehicle listing lets you search and manage your rental equipment assets. Clicking any Vehicle plate number launches the Vehicle Details Portal, featuring five core tabs:

  • Profile: View and edit model, brand, engine type, plate code, registration dates, and insurance policies.
  • Deployment History: View past projects and customers where this asset was deployed, including lifetime revenue.
  • Documents Checklist: Tracks physical scans of registration cards (Mulkiya) and local municipal permits.
  • Maintenance History: Lists all breakdowns, parts replaced, and preventive maintenance logs.
  • Expenses: Aggregates toll logs and fuel expenses charged to this plate.

Vehicle Details Portal Tabs

Clicking any Vehicle code launches the Vehicle Details Portal, featuring six tabs:

  • Profile: View and edit registration, chassis number, brand, model, insurance company, and license plate details.
  • Assignments History: Chronological list of all projects and customers this vehicle has been deployed to, including total hours and revenue generated.
  • Documents: Scans of the registration (Mulkiya), permits, and commercial licenses, with individual warning alerts for expiration.
  • Maintenance Log: Historical list of preventive services, breakdowns, parts replaced, and maintenance costs.
  • Fuel & Expenses: Tracks all Salik tolls, fuel logs, and miscellaneous expenses mapped to this vehicle.
  • GPS tracking: Real-time link to coordinate mapping of the asset (if telematics integration is active).

Step-by-Step Management:
  • Add a Vehicle: Go to /vehicles, click the "New Vehicle" button, fill in the Registration Number, select Type/Brand/Model, specify Ownership, Base Rent, Default Billing Cycle, and Daily Internal Cost, then click "Create Vehicle".
  • View & Audit: Click any vehicle plate number in the list to open its multi-tabbed Profile details (viewing Profile, Deployment History, Documents, Maintenance, and active Expenses).
  • Edit: In the details page, click the "Edit Profile" button to update rates or attributes.
  • Deactivate / Delete: Click the trash icon on the vehicle row or the "Delete" button in the detail view. Note: The system will run a referential integrity check. If the vehicle has active assignments or daily logs, deletion is blocked to protect billing data, and deactivation (setting to INACTIVE) is recommended instead.

5.2 Operator Profile Portal (`/operators`)

Manage your drivers, operators, and site crew records. Inside each Operator file:

  • Profile: Enter basic credentials, mobile number, basic pay rates, and bank details.
  • Compliance Schedules: Monitor visa statuses, medical checks, and Emirates ID expiry dates.
  • Qualifications: Upload scans of operator driving permits and heavy machinery licenses.

An operator with status INACTIVE or BLOCKED cannot be assigned to new blocks. If an operator is already on an active assignment and their status changes, the existing assignment continues generating DTLs — only new assignments are blocked.

Operators Visa & Compliance Schedules

Under Operators → Profile, the system features a specialized tab tracking legal compliance for multinational crews. The Visa & Compliance Schedule includes:

  • Labor Card Expiration: Tracks ministry card schedules and issue dates.
  • Emirates ID Expiry: Alerts the crew coordinator 90 days before physical card expirations.
  • Medical Test Schedules: Logs date of last medical test and schedules the next renewal, preventing illegal deployments.

Step-by-Step Management:
  • Add an Operator: Go to /operators, click "New Operator", fill in operator name, contact number, default Work Type, Hourly rate, Daily Internal cost, and passport/visa/license detail fields, and click "Create Operator".
  • View & Audit: Click the operator's name in the grid to open their details portal (viewing profile fields, upload checklists, and historical license compliance details).
  • Edit: Click the "Edit Profile" button inside the operator details view to update salary rates, statuses, or contact details.
  • Delete / Block: Use the "Delete" button to permanently remove if there are no historical logs. For active crew with logged time, update the status to INACTIVE or BLOCKED to restrict new assignments.

6.2 Deployment Blocks

Every resource in an assignment is a "block." A single assignment can have multiple blocks — e.g., 3 vehicle blocks + 2 material blocks + 1 operator block. Each block has its own date range (a sub-range within the assignment dates).

VEHICLE Block Fields

Specify the active vehicle, deployment start/end dates, base daily/hourly billing rates, overtime rules, operator requirements (with operator/dry hire), and whether weekends are included for time log auto-generation.

MATERIAL Block Fields

Specify the material item, deployment start/end dates, quantity deployed, and daily rate per unit.

Materials do not generate hours — they generate quantity-based DTLs (one per day with quantity = block.quantity).

DETOUR Block Fields

Allows deploying specialized bundled detour templates to assignments, pulling default crew and rates automatically.

When you add a DETOUR block, the system automatically creates child blocks for each resource in the template. A "2-Lane Detour" template requiring 2 vehicles, 2 operators, 10 cones, and 2 flagmen will create 6 child blocks instantly. You then assign the specific vehicles and operators.

Adding a Detour Block to an Assignment

When you add a DETOUR block and select a template:

System creates the parent DETOUR block referencing the selected template
System automatically creates child blocks for each resource in the template: 2 VEHICLE blocks, 2 OPERATOR blocks, 1 MATERIAL block (10 cones), 1 LABOUR block (2 flagmen)
You assign the specific vehicles and operators to the child blocks
All child blocks reference the parent DETOUR block via detourBlockId
Availability checks run on each child resource independently

6.3 Availability Checks

To prevent conflicts, the system automatically checks resource availability before finalizing an assignment block:

  • Resource Calendar Schedule: The system verifies that the selected Vehicle and Operator are not deployed on any overlapping active assignments for the proposed date range.
  • Quantity Capacity: For Materials and Labours, the system checks that the proposed deployment quantity does not exceed the remaining warehouse stock (Total Catalog Inventory minus currently deployed quantity).
Why Not Simple Date Range Overlap? A vehicle deployed Mon–Fri only is still available on weekends — date range overlap would incorrectly block it. The system generates the exact list of working days for the proposed block (respecting weekend exclusions) and checks each specific date.
Scenario A — Weekend Exclusion Vehicle A is deployed Mon–Fri (weekends excluded) from June 1–30. You try to deploy Vehicle A on a Saturday in June.
Result: ALLOWED — The first block excludes weekends, so the vehicle is genuinely free on Saturdays.
Scenario B — Full Week Conflict Vehicle A is deployed All Days from June 1–30. You try to add it anywhere in June.
Result: BLOCKED — Every calendar day in June is occupied.

6.6 Mid-Assignment Vehicle Replacement

When a vehicle breaks down or needs to be replaced mid-assignment, use the "Replace Vehicle" action on the block:

The current VEHICLE block is stopped: status → STOPPED, endedAt = today
A new VEHICLE block is created starting tomorrow with the replacement vehicle
DTLs 1 through today remain linked to the original vehicle block (preserved, immutable)
DTLs from tomorrow onward are generated for the replacement vehicle block
The timesheet for this period will show two lines for the same period — one per vehicle

The same "Change Operator" action applies when an operator needs to be replaced on a VEHICLE block.

6.7 Internal Assignments

An assignment marked isInternal = true does not require a customer. Used for company vehicles deployed for the company's own operations (e.g., office runs, site visits with company vehicles).

  • Generate DTLs normally — all automation works the same
  • Excluded from customer-facing timesheet generation by default
  • Do not produce invoices
  • Appear in utilization and internal cost reports

Daily Time Logs Listing Portal

A guide to the centralized registry where all daily times are logged and reviewed.

7.1 Finding and Filtering Time Logs

The centralized time log portal (located under `/time-logs`) stores every daily work hour. Coordinators can easily review operational activity using these key filter tools:

  • Calendar Date Range: Limit records to a single day, week, month, or custom period.
  • Client & Project Selector: Isolate work logs for a particular job site or customer.
  • Resource & Operator Filters: Filter to view activity for a specific operator or license plate.
  • Search Box: Instantly search by driver name, license plate number, or coordinator remarks.

7.2 Auto vs. Manual Logs

The system displays the source of each DTL:

  • Auto: The background scheduler generated this log at midnight based on the active assignment block settings.
  • Manual: Created manually by a coordinator using the "New Log" button to record an override or custom weekend deployment.

7.3 Hours Calculation During Auto-Generation

When the scheduler auto-generates a DTL, it calculates the hours breakdown based on the block configuration and company settings:

Example 1 — Standard Weekday, plannedOT = 2, fullDayHours = 8 workedHours = 8 + 2 = 10 | regularHours = 8 | overtimeHours = 2 | holidayHours = 0
Example 2 — Weekend Day (Friday), same block, includeWeekends = ON workedHours = 10 | regularHours = 0 | overtimeHours = 10 (all hours overtime on weekends)
Example 3 — Material Block (10 cones deployed) quantity = 10 | workedHours = 0 | regularHours = 0 | overtimeHours = 0

7.4 Standard Workday Flags & Multipliers

Each time log features simple visual badges summarizing operational variables:

  • W Weekend: Normal weekend day as defined in Company Settings. All worked hours on weekends are calculated as overtime hours.
  • H Holiday: Special holiday indicator toggled manually when creating or editing a log. Automatically applies the special holiday rate multiplier (e.g. 1.5x or 2.0x) to all logged hours.
  • Recording Absence or Leave: If an operator is absent or on authorized leave on a scheduled day, the system does not use special badges. Instead, coordinators manually edit the Daily Time Log's worked hours to 0.00 and record details (such as "Absent" or "Medical/Annual Leave") in the Remarks field, which successfully zeroes out regular worked hours and customer costs for that day.

7.5 Editing a DTL

Any DTL can be manually edited (via inline editing in the grid or the full edit form) to correct hours, flag a holiday, change the work type, or add remarks.

Editing a DTL does NOT automatically update any existing timesheet. You must regenerate the timesheet after editing DTLs to get the updated amounts.

7.6 The Snapshot Mechanism

When a DTL is created, it captures a snapshot of all rate data at that moment. The snapshot is immutable — subsequent rate changes have no effect on historical DTLs.

Why This Matters — Rate Change Scenario A vehicle's cost is AED 400/day in January. In February, it changes to AED 450/day. DTLs created in January still show AED 400. When you run a January profitability report in March, you see the correct January cost — not the current rate. Without snapshots, every historical report would be wrong the moment any rate changes.
Why This Matters — Name Change Scenario Operator "John Smith" changes his name in the system. All DTLs created before the change still read "John Smith" (snapshot). This ensures historical timesheets and audit trails remain accurate forever.

Timesheets — Deep Dive

A timesheet aggregates all daily time logs for a customer over a date range and calculates the billable amounts. It is the bridge between raw operational data (DTLs) and formal billing (invoices).

8.1 Generating a Timesheet

Required inputs:

  • Customer (or "Internal" flag)
  • Project (optional — filters logs to that project only)
  • Period Start and Period End

What happens internally, step by step:

System queries all daily_time_logs where: date is within period, isBillable = 1, assignment is ACTIVE or COMPLETED, and customerId/projectId match the selected criteria.
Groups logs into one line per resource per day. Grouping key: VEHICLE = date + vehicleId + operatorId; OPERATOR = date + operatorId; MATERIAL = date + materialId; LABOUR = date + labourTypeId.
Calculates calculatedAmount for each line using the billing formulas (see Section 6.2).
Inserts timesheet_lines rows linked to this timesheet.
Creates the timesheets header record with a unique sequential code (e.g., TS-00001).
Checks for overlapping timesheets: if another timesheet for the same customer/project already covers any part of this period (and is not INVOICED), the system warns. A force override allows proceeding.
LPO auto-fill: If the project has an LPO number and attachment, these are automatically copied to the new timesheet.

8.2 Cost Calculations & Billing Rules

To compile billing statements, the system aggregates all time logs (DTLs) within the timesheet date range and applies these clear, plain-English billing rules:

  1. Rate Normalization: Daily or monthly hire rates are normalized to an hourly baseline to compute partial days or overtime hours accurately.
  2. Vehicle Base Cost: Calculated by multiplying the total regular hours worked by the vehicle's normalized hourly rate.
  3. Operator Cost: If the vehicle block was scheduled "With Operator", the operator's hours are added to the billing. Standalone operator deployment lines are billed separately based on their specific contracted operator rates.
  4. Rate Multipliers: Regular overtime hours are multiplied by the overtime rate (e.g. 1.25x or 1.5x). Hours logged on official public holidays or designated weekends use the special holiday multiplier.

Full Worked Example — Vehicle with Operator, Weekday with OT

Settings: Full Day Hours = 8 | OT Multiplier = 1.5 | Holiday Multiplier = 2.0
Vehicle: baseRentAmount = AED 400 | baseRentType = DAILY | defaultRentCycle = DAILY
Operator: hourlyRate = AED 20
DTL: workedHours = 10, regularHours = 8, overtimeHours = 2 (plannedOT = 2)
vHourlyRate = 400 / 8 = AED 50/hr dailyRate = 50 × 8 = AED 400 vehicleCost = 400 + (10–8) × 50 = 400 + 100 = AED 500 operatorCost = (8 × 20) + (2 × 20 × 1.5) = 160 + 60 = AED 220 Total line = AED 500 + AED 220 = AED 720

Full Worked Example — Same Vehicle on a Weekend (Friday)

DTL: workedHours = 10, regularHours = 0, overtimeHours = 10 (all weekend hours = OT)
vehicleCost = 400 + (10–8) × 50 = AED 500 ← vehicle still charges daily + extra operatorCost = (0 × 20) + (10 × 20 × 1.5) = 0 + 300 = AED 300 ← all 10 hrs are OT rate Total line = AED 500 + AED 300 = AED 800

Full Worked Example — Material Deployment

Material: Traffic cones, costPerDay = AED 5/cone
Block: quantity = 10 cones, deployed for 3 days (3 DTLs created, one per day)
Each timesheet line: 10 × AED 5 = AED 50/day
Total for 3 days: AED 150

Monthly Rate Conversion Example

Vehicle: baseRentAmount = AED 15,000/month | baseRentType = MONTHLY | defaultRentCycle = DAILY | fullDayHours = 8
vHourlyRate = 15,000 / 30 / 8 = AED 62.5/hr dailyRate = 62.5 × 8 = AED 500/day IF worked exactly 8 hrs: vehicleCost = AED 500 IF worked 10 hrs: vehicleCost = 500 + (10–8) × 62.5 = AED 625

8.3 Outdated Detection

After a timesheet is generated, if any DTL that was included in it is later edited (its updatedAt timestamp changes), the timesheet shows an "Outdated" warning. This is your signal to regenerate the timesheet so it reflects the corrected hours.

8.4 Regenerating a Timesheet

If a timesheet is DRAFT or EXPORTED and allowRegenerationBeforeInvoice = true in settings:

All existing timesheet_lines for this timesheet are deleted
The system re-runs the full generation logic against current DTL data
New lines are inserted with current rates and hours
generatedAt and generatedBy are updated
Regeneration is permanently blocked if the timesheet is already INVOICED. To regenerate an invoiced timesheet, you must first delete the invoice (which reverts the timesheet to EXPORTED), then regenerate.

Invoices — Deep Dive

An invoice is created from a timesheet and represents the formal billing document sent to the customer. One timesheet = one invoice. The invoice captures the final billable amounts, VAT, adjustments, and LPO references.

9.1 Creating an Invoice

You can only create one invoice per timesheet. Navigate to a timesheet in DRAFT or EXPORTED status and click "Generate Invoice."

What happens internally:

Creates an invoices record with: sequential invoice number (e.g. INV-00001), date = today, due date = today + defaultDueDays, status = GENERATED, customer/project/period copied from timesheet.
Groups timesheet_lines into invoice_items. Lines with the same resource+operator combination are grouped; hours/quantities summed.
Calculates totals: subtotal = sum of all invoice_items; vatAmount = subtotal × (vatPercentage/100) if VAT enabled; grandTotal = subtotal + vatAmount.
If lockTimesheetOnCreate = true: sets the linked timesheet status to INVOICED.
Sends invoice notification email to customer if configured in Notification Settings.

9.2 Bundle Billing (Detour)

When a DETOUR block has bundleBilling = true, the invoice collapses all lines under that detour into ONE line item. The bundleBilling flag on a DETOUR block controls how the invoice presents the charges to the customer versus how the system logs the data internally.

Without Bundle Billing (bundleBilling = false) Customer sees: Vehicle 1 | Vehicle 2 | Operator 1 | Operator 2 | Operator 3 | Cones (10 units) | Flagmen (2 units)
= 7 separate invoice lines
With Bundle Billing (bundleBilling = true) Customer sees: "2-Lane Detour Service — 7 Days @ AED 850/day" = 1 invoice line
Internally: all 7 DTL streams still exist in full detail. The bundle is presentation-only.

Bundle mode only affects what the customer sees on the invoice. All individual DTLs, operator hours, OT tracking, and utilization data remain in full detail internally regardless of this setting.

VAT Formula

vatAmount = subtotal × (vatPercentage / 100)
grandTotal = subtotal + vatAmount + adjustmentAmount

Daily Automation & Email Notifications

An overview of how the system handles daily automated time log entries, automated assignment completions, and email updates in the background.

13.1 The Daily Automation Cycle

RentERP runs automated background tasks at midnight relative to your local timezone to reduce coordinator manual entry workloads. These automations require zero user intervention:

1
Auto-Generate Daily Logs (DTLs)

Every night at midnight, the system scans all active assignments. If today is scheduled for deployment (and weekend rules match), it automatically creates a draft time log pre-populated with standard regular shift hours.

2
Auto-Complete Assignments

If an assignment reaches its scheduled end date, the system automatically transitions its status to COMPLETED, releasing the vehicle and operator back to the available pool for future projects.

3
Compile & Dispatch Expiry Alerts

The system runs compliance checks and compiles expiring credentials (visa cards, driving permits, vehicle registrations) into a unified morning email report sent to the crew coordinator.

13.2 System Email Notifications

The system dispatches formatted email notifications during core operational events:

  • New Assignments: Dispatched to coordinators or operator contacts upon assignment activation, listing project details and start times.
  • Timesheets for Approval: Sends a PDF timesheet link to the customer contact, allowing them to review and approve hours digitally.
  • Invoice Dispatch: Emails the client contact a PDF copy of finalized invoices alongside secure payment terms and instructions.

13.3 Scheduler Technical Details

Alignment Logic The scheduler aligns to interval boundaries, not rolling delays. If frequency = 5 minutes and it's currently 12:03, the scheduler waits until 12:05, then 12:10, 12:15, etc. This prevents time drift over long periods.
Resilience If any tick throws an error, the scheduler catches it, logs the error, and retries in 1 minute (fallback heartbeat). It never crashes permanently. A single failed tick does not prevent subsequent ticks from running.

13.4 Manual Trigger

Administrators can manually trigger the daily automation cycle from the system dashboard. This is useful after correcting DTLs or recovering from a missed scheduler window.

Manually triggering a job is safe — it is subject to the same idempotency guards as the scheduler. Running "Generate Daily Time Logs" manually will not create duplicate DTLs if they already exist for today.

13.5 Timezone Handling

The scheduler uses the timezone configured in Settings → Company to determine "midnight" for DTL generation. This ensures all time log dates match your local calendar, not the server's clock.

Why Timezone Matters If the company is in Dubai (UTC+4) and the server runs on UTC, without timezone correction the scheduler at 11:30 PM UTC thinks it is still "today" — but the Dubai calendar already shows the next day. The timezone setting ensures DTLs are generated for the correct date as experienced by the operations team.

RentERP uses a role-based access control system. Every API route checks the user's permissions before processing. A user without the required permission receives a 403 Forbidden response.

  • A role assignment (one role per user)
  • Status: ACTIVE or locked
  • Account lock: After repeated failed login attempts, the account is temporarily locked. An admin can unlock it from the user detail page (Users → Unlock). The primary Super Admin account cannot be unlocked this way — see the FAQ chapter for the database recovery command.

Scenario Library — Q&A

Real operational questions answered step by step. If you encounter a situation not covered elsewhere in this manual, check here first.

A vehicle is deployed Mon–Fri (weekends excluded). The customer asks for work on a Saturday. Can I create a DTL for Saturday?

Yes. Create a manual DTL from the Time Logs module for that vehicle on the Saturday date. The scheduler would not have generated one (because includeWeekendsForAutoLogs = false on the block), but manual creation is always possible for any date within the block's start–end range.

The Saturday DTL will automatically have isWeekend = 1. All hours will be classified as overtime (regularHours = 0, overtimeHours = all worked hours). The timesheet will correctly charge weekend rates.

I have a 30-day assignment. After 10 days, the vehicle breaks down and needs to be replaced. What is the correct workflow?

Use the "Replace Vehicle" action on the VEHICLE block:
  1. The current block is stopped: status → STOPPED, endedAt = today
  2. A new block is created starting tomorrow with the replacement vehicle
  3. DTLs 1–10 remain linked to the original vehicle (preserved, immutable)
  4. DTLs 11–30 will be generated for the replacement vehicle
  5. The timesheet will show two lines for the same period — one per vehicle
The availability system automatically frees the original vehicle from day 11 onward.

I generated a timesheet but the hours are wrong for 3 days. What is the correct workflow?

  1. Go to the Time Logs module and edit the 3 incorrect DTLs (correct the hours)
  2. Navigate back to the timesheet — it will show an "Outdated" warning
  3. Click "Regenerate Timesheet" — all lines are deleted and recalculated
  4. Verify the new amounts are correct
  5. Proceed to invoice
Important: If the timesheet is already INVOICED, you must delete the invoice first (which reverts the timesheet to EXPORTED), then regenerate.

A customer has two projects running simultaneously. How do I generate separate invoices for each project?

When creating each assignment, set the projectId to the appropriate project. When generating timesheets:
  • For Project A: Select customer + Project A → timesheet TS-XXXXX includes only Project A DTLs
  • For Project B: Select customer + Project B → timesheet TS-YYYYY includes only Project B DTLs
Each project gets its own timesheet code and its own invoice. LPO numbers per project are also auto-populated separately.

What happens if I try to assign a vehicle that is already deployed on those dates?

The system blocks the save with an availability conflict error. The error message identifies which existing assignment has the vehicle occupied on those specific dates. Your options:
  • Choose different dates for the new block
  • Choose a different vehicle
  • Stop the conflicting block on the existing assignment (if the vehicle is no longer needed there)

A vehicle is deployed Mon–Fri for 4 weeks. How many DTLs are created, and what do they contain?

4 weeks × 5 working days = 20 DTLs. The scheduler skips Saturday and Sunday because includeWeekendsForAutoLogs = false.

Each DTL (assuming no planned overtime and fullDayHours = 8):
workedHours = 8 | regularHours = 8 | overtimeHours = 0 | isWeekend = 0

We have a detour with 2 vehicles and 3 operators. If bundleBilling is OFF, how many invoice lines does the customer see?

One line per resource: Vehicle 1, Vehicle 2, Operator 1, Operator 2, Operator 3 — plus any materials and labour in the template (e.g., cones and flagmen). If the template has 10 cones and 2 flagmen, that is 7 invoice lines total.

If bundleBilling is ON: exactly 1 invoice line — "2-Lane Detour Service — X Days." All 7 resource streams still exist as DTLs internally.

A material (e.g., cones) has 10 total quantity. Can I split them across two simultaneous assignments?

Yes — as long as the total requested across all active blocks on any given day does not exceed 10.
  • Assignment A: 6 cones + Assignment B: 4 cones = 10 ≤ 10 → ALLOWED
  • Assignment A: 6 cones + Assignment B: 5 cones = 11 > 10 → BLOCKED
The check is per-day — if the split only overlaps for some days, the system checks each day individually.

Can I invoice the same timesheet twice?

No. The system enforces a one-to-one relationship between timesheets and invoices. Once a timesheet is INVOICED, the "Generate Invoice" button is removed. If you need to re-invoice, delete the existing invoice (which reverts the timesheet to EXPORTED status), then generate a new one.

If I change an operator's hourly rate, do existing timesheets and invoices change?

No. Existing timesheet lines use the rateSnapshot captured at the time the DTL was generated. The new hourly rate only affects DTLs created after the change. This is the snapshot mechanism — historical data is immutable and remains accurate regardless of any future rate changes.

The profitability report shows a negative margin on one timesheet. What does that mean?

It means the billed amount is less than the estimated internal cost. Common causes:
  • The billing rate was set lower than the vehicle's costPerDay
  • Many overtime hours inflated the operator's cost, but the vehicle was billed at a flat daily rate that did not account for OT
  • Rate configuration error — billing rate set incorrectly when the assignment was created
Fix the underlying rate settings and issue a revised invoice (delete current invoice → adjust → re-generate).

What is the difference between stopping a block and deleting it?

  • Stopping a block: Sets status = STOPPED, records endedAt. All existing DTLs for this block are preserved. The block remains visible in the assignment for historical reference. No more DTLs are generated from the stop date onward. This is the correct action when a vehicle breaks down or a resource is removed from service mid-assignment.
  • Deleting a block (removing it during an assignment edit): Deletes the block AND all its associated DTLs AND all child blocks (for DETOUR blocks). Historical time logs are permanently and irrecoverably lost. Use only when the block was created by mistake and has no real operational history.

Can a timesheet span multiple months?

Yes. The period start and end dates can cross month boundaries. All DTLs within the specified range are included in one timesheet. There is no system restriction requiring a timesheet to fit within a calendar month. Many operators generate monthly timesheets by convention, but this is not enforced by the system.

How does the system handle public holidays? Can I mark a specific date as a holiday?

There is no standalone holiday calendar UI in the current version. Holidays are handled at the DTL level. When editing a DTL manually, you can toggle isHoliday = true, which causes that day's hours to use the holidayMultiplier instead of the overtime multiplier.

The scheduler does not automatically detect public holidays — holiday marking is a manual operation on individual DTLs. Mark holidays in DTLs before generating the timesheet, or regenerate the timesheet after marking.

The scheduler is enabled but no DTLs are being generated. What do I check?

Work through this checklist in order (see Section 13.3 for technical details):
  1. Is cron_settings.is_enabled = 1? (Settings → Cron)
  2. Is the assignment status ACTIVE?
  3. Is enableAutoTimeLogs = true on the block?
  4. Is block.status = 'ACTIVE' (not STOPPED)?
  5. Is today within the block's startDate to endDate?
  6. Is today a weekend AND includeWeekendsForAutoLogs = false?
  7. Does a DTL already exist for today for this block? (Already generated correctly)
  8. Check the cron run history for errors (Settings → Cron → Run History)

What triggers the "Cannot assign vehicle — active assignment conflict" error even when the dates appear to be different?

This happens because availability uses occupied-date logic, not simple date-range overlap. The system generates the exact list of working days for the proposed block (respecting the includeWeekendsForAutoLogs setting) and checks each day individually.

Example: An existing block runs June 1–30 with ALL days included. Your new block runs July 1–3, but somehow overlaps. This would not conflict. However, if a block runs June 25 – July 5 (All Days), and you propose July 1–3, that would conflict on July 1, 2, 3.

Check the existing block's weekend inclusion setting, and verify that no "All Days" blocks straddle the dates you are trying to use.

Can an assignment be ACTIVE with all blocks stopped?

Yes. The assignment status (ACTIVE/COMPLETED) is independent of the individual block statuses. If all blocks are STOPPED, no new DTLs will be generated — but the assignment itself remains ACTIVE until its end date passes (when it auto-completes). You can still add new blocks to an ACTIVE assignment at any time.

How does the expense code "EXP-0021" get generated? Why does it start at 21 and not 1?

The starting number is configurable in Settings → Expense Settings. The system was set up with a starting number of 21 — typically because the system continues numbering from a legacy system that already had expenses 1 through 20. Each new expense increments atomically from the current counter, so there are never gaps or duplicates.

What happens if SMTP is not configured and the system tries to send an email?

The email operation fails gracefully. The underlying record (assignment, timesheet, invoice) is still created and saved successfully. Only the email send fails. The failure is logged internally but does not surface as an error to the user in a blocking way, and does not roll back the main database operation.

How do I unlock an administrator account if they get locked out of the system?

If an employee is locked out of their account due to repeated failed login attempts, any administrator can log in and navigate to Settings → Security → Users (or standard Users list). Find the locked operator in the list and click the active Unlock Account button next to their name.

This instantly clears their lockout penalty and lets them log in again immediately.

How do I perform a manual database backup and restore via the system settings?

Administrators can manage database backups directly in Settings → Backup & Restore.

Backing Up: Click the "Generate Backup" button. The system automatically compiles all records, tables, and settings into a secure, compressed backup file stored in your cloud storage.

Restoring: To restore a historical backup point, select the desired file from the backups list and click "Restore". The system will ask for confirmation and then reload the system to that exact historical date.

WARNING: Restoring a database replaces all current data. All active users will be logged out to prevent data synchronization conflicts.

How do I test my newly configured branding customization?

Once you save a custom brand theme (e.g., uploading your logo, choosing dark/light primary colors, setting slogans) under Settings → Branding, the changes are applied globally and instantly to all active users.

To verify branding changes:
  1. Clear your browser cache or log out and log back in.
  2. Inspect sidebar placements to ensure custom logo ratios are correct.
  3. Send a draft timesheet to your own email to confirm that email headers use your primary hex color and brand name correctly.

Admin Tools: Backup, Calendar & Recovery

A guide to administrator-only tools: system backup and restore, the assignment calendar, DTL backfill for missed logs, and the password reset flow.

16.1 Backup & Restore

Administrators can take a complete snapshot of the entire system from Settings → Backup & Restore. This requires the Settings: Edit permission.

Taking a Backup

Click Generate Backup. The system produces a ZIP file (backup_{company}_{date}.zip) containing:

  • database.sql — a full MySQL dump of all tables and data
  • files/ — all uploaded documents (vehicle registration scans, operator compliance files, invoice attachments, etc.)
Backup is streamed directly to your browser — it is not stored on the server. Save the file securely. The backup action is logged to the audit trail.

Restoring from a Backup

Upload a previous backup ZIP and select which components to restore:

  • Database only — restores all data; uploaded files remain as-is
  • Files only — replaces the uploads directory; database unchanged
  • Both — full system restore to the backup point
WARNING: Restore is destructive. Restoring the database drops and recreates the entire tenant database. All data created after the backup point is permanently lost. All active users will be logged out immediately. Only restore when you are certain.

16.2 Assignment Calendar View

The Calendar view (Assignments → Calendar) shows all deployment blocks as a visual timeline across a selected date range. It requires the separate Assignment: Calendar View permission.

  • VEHICLE, OPERATOR, MATERIAL, LABOUR, and DETOUR blocks all appear as color-coded events.
  • Maintenance windows (SCHEDULED and IN_PROGRESS) are overlaid on the same calendar, so coordinators can immediately see conflicts between active maintenance and deployment blocks.
  • Fleet totals (active vehicles count, active operators count) are displayed in the header.
  • Clicking an event navigates to the full assignment detail.
Calendar View ≠ Assignment: View Granting a user Assignment:View lets them see the assignment list and details. They still need the explicit Assignment: Calendar View permission to access the calendar page.

16.3 Generate Missing Logs (DTL Backfill)

If the scheduler missed a day (e.g., server downtime, or the cron was disabled), administrators can backfill all missing DTLs for any assignment from the assignment detail page using the Generate Missing Logs action. Requires Settings: Edit permission.

How it works:

  • Iterates every block in the assignment from its startDate through today.
  • Checks each date: if a DTL already exists, it is skipped (fully idempotent — safe to run multiple times).
  • Skips STOPPED blocks and blocks with enableAutoTimeLogs = false.
  • Skips weekends if the block/company settings exclude them.
  • Creates backfilled DTLs with remarks = "Auto-generated backfill missing logs".
  • Returns a summary: { createdCount, skippedCount }.
Safe to Re-run Running Generate Missing Logs on an assignment that already has all its DTLs will create 0 new logs and skip all existing ones. There is no risk of duplicates.

16.4 Password Reset Flow

Users who forget their password can use the Forgot Password link on the login page.

The flow:

  1. User enters their email address.
  2. System checks if the email exists and sends a secure reset link (JWT token, signed with HS256, valid for 30 minutes).
  3. User clicks the link → redirected to the Reset Password page.
  4. User enters and confirms a new password. New password is saved (bcrypt hashed).
Anti-Enumeration Protection The system always responds "If that email exists, a reset link has been sent" — regardless of whether the email is registered. This prevents attackers from discovering which email addresses are in the system.
Rate Limiting Repeated reset requests from the same email + IP address are throttled. If you see HTTP 429 "Too many reset requests", wait and try again. The retry window is included in the response header.
All password reset events (requested, sent, completed) are logged to the tenant audit trail.
